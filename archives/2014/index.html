开始响应式布局之旅

响应式布局

  1. 优雅降级

    “优雅降级”:当你利用所有新的特性(那些能力最强的浏览器最强的特性),你要保证那些相对较老的浏览器也能访问你的内容.

    优雅降级不见得是未来友好型的,它表现出了一种对使用旧的浏览器的用户不尊敬,而且忽视了现实状况,因为还有很多崭新设备(手机)也只带有能力较弱的浏览器.

    “渐进增强”:从本质上来说,就是把优雅降级给倒了过来.不像优雅降级那样为先进的,最强进的浏览器开发站点,然后让功能差劲的浏览器也能使用多少功能.相反,渐进增强会先创建一个基本体验,该基本体验使用具有语义的标记和结构定义,并侧重于让内容以一种简洁的方式来展现,同时这也是一种为能力更强的浏览器创建更丰富的体验.

    把渐进体验比喻为一颗花生豆.内容是花生,样式部分是(css)是巧克力,交互部分(javascript)是糖衣.虽然内容部分可以单独存在,但是在最外面加上功能层后,体验就会变得更加丰富而完整了.

    响应式设计使用了同样的思想,并会针对不同的设备提供适当的内容和布局.即先从一个基本的布局开始,然后使用流动布局和媒介查询这些技术来为功能更多,屏幕更大(功能更多和屏幕更大并不总是同义的)的设备增强使用体验.

  2. 流动布局

    我相信,web最大的有点常常被人们视为是限制和缺陷.灵活性是web固有的本性.作为设计师和开发者的我们,应该去拥抱它的这一特性,并且要设计开发出同样具有灵活性的页面,使得所有的设备都可以访问.

    作为灵活性的第一步,就是要为我们的站点创建流动布局,并藉此来对不同尺寸的设备屏幕做出不同的响应.

    布局:4种不同的布局.

    固定布局: 页面宽度会被指定为特定大小的像素.其中960像素是使用最为广泛的宽度.固定布局是web布局中最为常见的实现方式,因为固定布局会给人较多控制权的错觉.能确切知道你的站点会以多宽来显示,这使得你能创建出拥有大量的图形的设计,而且即使是在不同的屏幕上,它们也看起来相对一致.

    缺点: 严格的固定布局在今天广泛而多样的设备生态系统中早已问题重重,因为当今很多最新,功能最强的手机和平板电脑的浏览器会默认显示缩小之后的网页,以此来适应它们那较小的屏幕.虽然可以通过手指来缩放页面,但是这样的操作会让人感到烦琐.

    流动布局: 度量的单位不是像素,而是百分比,这样可以使页面具有可变的特性.

    在实施诸如媒介查询,针对不同的分辨率优化样式这样的响应式策略时,流动布局能使这些策略更加容易实现.由于没有那么多的问题需要修复,因此需要写的CSS样式也就相对较少.

    单独使用流动布局这一方法依然不足以保证所有元素在从手机到电脑屏幕上等一系列设备都能保持良好的效果.因为有些文本的行宽在大屏幕上会太宽,而在小屏幕上又看起来太窄.

    流动布局仅仅是个开始.

  3. 弹性布局

    弹性布局与流动布局类似,只是他们的度量单位不同.通常情况下,在弹性布局中会以em来作为单位.1em相当于当前字体的大小.如果body的字体为16px,那么1em就等于16px.

    但是在弹性布局中也可能出现令人讨厌的水平滚动条.如果你把字体大小设置为16px,并把容器宽度设置为55em,那么就会在任何宽度小于880px的屏幕中国出现水平滚动条.弹性布局中这个问题比固定布局中的情况还要有更多的不稳定性,因为如果用户把字体放大到了18px,那么你的容器就便会达到990px.

  4. 混合布局

    混合布局结合了上面提到的两种或两种以上的布局方式.

    假设你有一个300px宽的广告区域,那么你可以将放置广告的边栏指定为固定的300px,其余各列的宽度则以百分比为单位.这样既保证了广告的宽度为固定的300px(考虑到第三方广告服务,这样的安排很有必要),同时除边栏之外的其他内容也能填满剩下的整个空间.

    但在流动布局中使用上面方法很容易让页面混乱不堪.如果边栏设置为300px,把主页内容设置为70%,那么当屏幕小于1000px时,还是会遇到水平滚动条问题.

  1. 问题,那种布局最具有响应性????

    根本上来说,者取决于你特定的项目,因为每一种方式都有其优势和缺点.

    大多数情况下,最佳答案是具有灵活性的那几种布局:流动布局,弹性布局或者混合布局—-因为相对固定布局,他们对未来更加友好.

    一句话,没有完美的答案.

  2. 字体大小px

    为了让设计拥有web的流动性,那就意味着你的设计中能灵活改变字体大小,可以使用任意单位来设置字体大小,但是主要方法无外乎三种: 像素,em和百分比.

    像素:一直以来,像素都是人们喜欢的字体大小单位,无论如何设置字体大小,你都能对其进行准确的控制,如果你把它设置为18px,那么每个浏览器都会将其设置为18px.

    但是这样的掌控会付出一定代价.虽然使用像素作为字体的大小单位不需要考虑级联—-父元素的字体大小对子元素没有影响,但也意味着当你想让某些文本以不同的大小显示时,就需要对他们逐一设置.这对于维护来说,非常不利.你就不得不逐一修改每一个值.

    以像素为单位的字体在老版本的IE中是无法实现缩放的,意味着使用IE9以前的,字体默认大小的IE浏览器的用户是不能调节你的页面中的字体大小的.

    用像素指定字体大小依然不是对未来特别友好的方式,使用像素作为字体大小的单位,是与web灵活性原则背道而驰的做法.

  3. 字体大小em

    使用em作为字体大小的单位是一种更加流行,具有灵活性的方法.1em等于默认的字体大小.

    em可跨浏览器进行缩放,而且他们也是级联的.这即可以是好事情,也可以是坏事.从简化维护的角度来说这是好事,因为这样可以相对地指定元素大小,意味着你只需要调整初始化的基准,其余部分就会自动调整,而且按比例调整.

    但级联也会让事情变得复杂.

    要试着结构化你的css和html,并保持字体的可预测性.如果你的内容部分使用了基准字体大小,只是修改了标题元素大小,那么你就可以避免这些问题了.

  4. 字体大小:百分比

    和em一样,以百分比为单位的字体也是可以缩放的,而且是级联的.与em相似,如果字体大小是16px,那么100%就相当于16px,200%相当于32px.

    从理论上来讲,百分比和em没有很多的区别,而em逐渐成为了web中更受欢迎的字体度量单位.其实没有什么技术原因.

    IE浏览器,却在显示em的时候有问题.如果基准是以em为单位的,那么IE会夸大调整字体的大小.如何解决ie夸大问题:

    你可以在body中加入百分比为单位来指定基准字体大小:

    body { font-size: 100%; }

    在大多数浏览器中,默认的字体大小都是16px.通过Body的字体设置为100%后,你就可以保证页面是可以缩放的,而且不会有夸大现象.之后,你就可以使用em来指定字体大小了.

    桌面浏览器默认的字体大小是16px/

    但是,这一特性对于其他类型的设备有时就不那么奏效了.比如黑莓6.0系统上,默认的font-size是22px,Kindle Touch默认的字体是26px.

    很多新的设备都有较高的分辨率,因此16px的字体在上面会看起来很小.大多数设备只好向浏览器说,自己有一个与实际不符的分辨率来绕过问题.

  5. 字体大小: 奖励关卡: rem

    rem: root em

    rem与em的区别: rem大小与根像素—HTML元素有关.使用rem可以避免发生在嵌套元素中的级联问题.

    通常情况下,rem在桌面浏览器得到了很好的支持.IE9+,Chrome6.0+,Safari5.0+以及Opera11.6+都支持rem.另外,IOS4.0+和Android2.1提供了相应支持.但opera mobile还都不支持.

    为了支持上面情况,你可以额外设置一个以像素为单位的备用方案.

    span { font-size: 16px; font-size: 1rem; }

  6. 字体: 哪种是最具有响应式的??

    使用em不但可以使字体能够缩放,而且维护起来也更加容易.

  7. 网格布局

    相对于css布局,表格布局的优势之一在于它简化了将站点布局为多列的实现过程.可以混合使用固定宽度和流动宽度: 一行里可以使用包括好几列.

    `.main {

    display: table-cell;

    }`

    是时候扫除css表格值的尘土,开始使用,移动平台对此支持也很好. 如果将某列设置为table-cell,那么就可以使用固定宽度和滚动宽度了.

    display: table;告诫:

    在一个被你指定为display: tablecell;的元素内,你无法精确匹配定位元素.如果你需要精确定位,你就不得不在表格系统中再插入一个div,或者干脆不使用display: table.

    web设计中没有银弹:<银弹是指的是那些解决复杂而棘手的问题方法或者技术手段>,仔细考虑你的需求,包括使用display: table;.

    记住,站点不需要在不同设备的不同浏览器中看起来一模一样,事实上也无法做到的.

  8. 布局总结

    大多数情况下,流动布局(以百分比为单位,因而可以根据屏幕来调整自身布局)是你设计站点时的最佳选择.当宽度被字体大小限制时,可以使用弹性布局,当宽度被百分比限制时,你可以使用流动布局.

    我们要坚持使用百分比和em,尽管rem未来很有潜力.

    通过使用CSS表格,你可以很容易的将固定宽度与流动宽度结合使用,现代浏览器都有极为出色的支持.而且你还可以通过条件注释来为IE7以及以下版本提供备用样式.

  9. 媒介查询

    媒介查询可以让你根据在特定环境下查询到各种属性值—-比如分辨率,色彩深度,高度和宽度—-来决定采用什么样的样式.

  10. 视口

    在桌面浏览器中,视口是一个简单概念,视口就是浏览器的可视区域,也指浏览器的宽度.

    像素就是像素,除非它不是像素.

    浏览器涉及到两种像素: 设备像素和CSS像素.

    你需要在视口中考虑两种不同的像素. 你有两种不同的视口要考虑: 布局视口和视觉视口.

    布局视口与设备像素非常相似,因为他们都度量单位是一样的.但是视觉窗口是可以变化的,视觉窗口指的是屏幕上显示的那一部分页面.在移动窗口上,这便成为了一件复杂的事情.

    webkit通过视口元标签,我们可以控制页面在很多设备上的大小和布局视口.

    <meta name="viewport" content="directiv, directive"/>,该标签需要放在head标签中.width声明可以让你将视口设置为某一特定的宽度,或者为设备屏幕的宽度.

    <meta name="viewport" content="width=device-width" />在width声明中使用device-width是最佳选择.

    你最好不要为width设置宽度,而要使用device-width.与width对应的是height,当然可以这样声明:<meta name="viewport" content="height=device-height" />高度.在实践中,你也许并不会指定高度.因为height唯一有用的地方,就是当你不想让页面垂直滚动的时候,但是这种情况并不多见.所以,你可以不使用height.

    css设备配对:实际上,视口元素不是规范的.

    @viewport可以让你在CSS中直接使用视口元标签中的那些描述符(width,zoom,orientation,resolution等).譬如:

    `@viewport {

    width: device-with;

    }`

  11. 媒介查询

    媒介查询可以通过询问浏览器来确定特定的表达式是否为真.如果是真,那么就会直接加载一些特殊的,适用于这种情况的样式,从而达到调整布局的目的.

    媒介查询的一般形式:

    @media [not|only] type [and] (expr) { rules }

    一条媒介查询包含如下信息:
    媒介类型: 特定目标的设备
    媒介表达式: 测试某一特性是否为真
    逻辑关键词: 你可以使用关键词(and,or,not)来测试出更复杂的表达式.
    规则: 调整显示效果的基本样式

  12. 媒介类型

    css定义了10种不同的媒介类型. all(所有设备,默认).媒介查询的强大地方就在于他们利用表达式来检测设备不同特性的真与假.比如

    `@media screen and (min-width: 320px) {

    }`

    这条语句可以来检测出两种东西: 检测页面是否属于screen,然后再测试设备的视口宽度.其中min会保证视口宽度至少为320px.

    很多较老的浏览器支持媒介查询,但并不支持媒介查询.有些甚至会导致浏览器尝试下载那些你不希望让用户看到的样式.这时你可以使用only关键字,为那些较老的浏览器隐藏媒介查询.譬如:

    @media only screen and (color)

    媒介查询可以写在页面内部,当然也可以通过link元素的media属性被包含到页面中来.

    `a {
    text-decoration: none;
    }
    @media screen and (min-width: 130px) {
    a {

    text-decoration: none;

    }
    }`

    到底使用那种方案,取决于你的项目,因为这两种方案都有利弊.

    内嵌媒介查询中,无论是否使用到,所有的样式都会被下载起来,但其好处是只需要一个HTTP请求即可.移动网络上一次HTTP请求所需要的时间可能是有线连接的4到5倍.当然内嵌样式也有缺点,那就是单个CSS文件会变得很大.

    因此,在你节省下一些请求数的同时,你也创建了一个较难维护的大型文件.

    外部媒介查询的优势在于文件可以变得很小,进而利于维护.

    我推荐人们使用内嵌媒介查询,因为额外的http请求肯定会让站点变慢,而且性能是如此重要,以至于我们不能忽视掉他们

  13. 媒介查询顺序

    从桌面端向下设计的,因为他的默认布局是你在笔记本或者台式机的浏览器所看到的那样.这种思想的通常设计.

    /*base design*/ @media all and (max-width: 768px) { ... } @media all and (max-width: 320px) { ... }

    但是向下设计会造成一些问题.虽然目前移动设备对媒介查询的支持有所改善,但是依旧不够完善.比如黑莓(6.0以前版本),Window Phone 7还有NetFront(驱动着前三代kindle)对媒介查询都缺乏必要的支持.

    每位用户都拥有最新最好的设备—-这样的想象虽然很有趣,但事实并非如此.Android4是最新的设备,但是有将近66%的设备依旧运行着Andirod2.3x或更低的设备.同样的问题在黑莓中依然存在——目前有66%的黑莓用户对媒介查询缺乏支持.

    事实上,并不是每个人都追赶快速发展的新技术,而另外一些人或许压根就买不起新的技术和设备.

    从移动端向上设计:

    优先建立移动体验确保那些不支持媒介查询的移动设备也能获得一个合适的布局.你唯一需要着手处理的就是IE.因为IE9以前的浏览器器是不支持媒介查询的,当然这个问题也可以解决.

    一个采用移动端向上的样式表通常是这样的.

    @media all and (min-width: 320px) { .... } @media all and (min-width: 768px) { ... }

    优先创建移动体验还可以降低CSS的复杂性.

    创建核心体验:覆盖面广是互联网伟大特性之一,要尽可能发挥它的优势.

    确定断点:传统的确定断点都是使用一些固定的标准宽度:320px(iphone和其他一些设备),768px(ipad)和1024px.

    当你完全根据常见设备来确定断点的时候,你就会冒着为这些特性宽度开发而忽视这些宽度之间(例如iphone横屏后为480px)的设备的风险.同样这种方法对未来也不是友好的.你将不得不增加新的断点.

    增强对大屏幕的支持.设置max-width来设置窗口浏览器窗口的最大宽度.或者通过字体来增大来增加行高.

    使用em来为媒介查询增加灵活性.

    人们会使用不同缩放级别的浏览器来浏览web,有些视觉障碍的人也许会觉得很多站点的文字难以看清,因此会将默认的缩放级别增大.在firefox和opera中,这不是什么问题,因为基于像素的媒介查询将会按照缩放级别重新计算样式,并应用到字体上.但其他浏览器中,我们之前一直很完美的断点开始让我们失望了.

    我们可以使用em来解决这些问题,而让我们的站点更加灵活.

  14. 导航栏

    导航栏是我们总结媒介查询前处理的最后一个元素.如果用户不知道如何前往他们想去的页面,那么即使站点有再好的内容也不会留住用户.

    无论用户使用的是什么样子的屏幕,你的导航栏都应该是可以访问的,而且要易于使用.

    我们期望的导航栏:
    它不应该占用宝贵的屏幕空间
    它应该是直观的,这样用户就不会迷失方向或者感到困惑
    他应该在各个设备上应用(尽管体验会根据所使用的设备能力的不同而不同).

    只有IE9以及更高版本的IE才支持媒介查询

  15. 媒介查询结束

    智能手机总是希望能为人们提供完整的web体验,如果不使用视口元标签,那么大部分智能手机都会显示一个缩小版的站点.

    媒介查询让我们坚持诸如视口宽度或高度这样的特性,并据此来适当地调整CSS样式.

    在媒介查询中,使用em来替换px,可以让响应式站点更加灵活而且是可访问的.

    一定要在真实的设备上测试,这样可以提醒你应该针对不同的屏幕对类似于导航菜单这类组件做出相应的调整.

    display: none;不能在性能上有任何的帮助.只是简简单单地隐藏掉了,DOM元素依旧会被创建,隐藏的资源依旧会被下载.

    默认浏览器行为:先使用默认方法来下载小屏幕使用的图片,然后再在需要的时候下载比较大的图片.但是缺点是,大屏幕设备得在只需要发送一次请求的地方发送两次请求,所以尽可能避免这种情况,大屏幕上设备也同样需要.

    使用第三方解决方案的缺点.如果公司改变了它的现有规则或者停业了,那么你就可能会被冷落,并且不得不再去寻找另外一套崭新的解决方案.

    不是要隐藏内容,而是要加强内容.要更加严肃对待内容.要进一步优化用户的用户体验.

    拥有触摸屏并不意味着能够支持触摸事件.如果不支持触摸事件,就降级触发点击事件.要仔细考虑设备的各项能力,以及如何针对他们做出相应的定制.通过删除组件和样式,你可以轻易地为触摸屏幕,低端设备以及具有电话功能的设备做优化.

如果你足够幸运,圈外的人也会听到你的大名,比如Linus Toralds或者linux.

仅仅比你认识的软件开发者好,并不能使你成为卓越的开发软件师.仅在量上超过其他人,是不足以让你的声誉像病毒一样扩散开来的.

怎样做才能使你自己和你的工作成绩把变得像那头紫色的奶牛一样引人注目呢?对一门学科来说,不仅仅是掌握它—-可以撰写一本关于此科目的书籍.

但是事实上,非常出色的人是不会介意有人想要认识他们的.

持久的联系就是持久的联系.

很多软件开发行业里最具有影响力的想法都可以通过邮件甚至是线上即时交谈获得.

  1. 保持技术领先

如果你不随时保持警惕,这种危险就会悄然而至,给你来个措手不及.当你发现的时候,已经晚了.

你掌握的知识越是主流,过时的危险也就越大

一件事情是可以确定的,那就是任何事情都处于变化之中.

不仅糟糕的过程会制造出糟糕的产品,糟糕的产品也会导致糟糕的过程.

重点是你穿越这条道路的过程,而不是这条道路的终点.

洗碟子是一项单一的工作,没人愿意做.软件开发师每天也经历类似单调沉闷的工作,例如事件追踪和费用报告.下次再从事类似工作的时候,不要焦急地想要赶快做完它,尝试在工作中关注任务本身.

没有蓝图,你可能会迷失方向.

市场在不断变化着,稍不留意就会赔钱或者错失赚钱的机会.

一名良师或者关系好的同事不是你肚子里的蛔虫,所以可以比较客观地对你做出评论.

别让过时悄悄地降临到你的身上,就像你的裤子突然之间就变得紧身了一样.

独立不是一件简单的事.这把你所有的技术作为一个专业来测试.现在你可能没有做好准备.但你也没必要做足这一切.把它当做个人发展项目,在业余时间去推销自己.

所以,什么东西是令你真正的好奇呢?试着跟随自己的兴趣,看看会发生什么?结果或许让你大吃一惊.

css3实用指南阅读笔记

写在前面

尽可能广泛的受众提供尽可能优质的体验

仅仅因为部分人不愿意或者不能升级浏览器,却让使用新型浏览器的用户无法享受CSS3所提供的伟大技术,这是毫无道理可言的

更少的标记和图片意味着用户需要下载的内容会更少,页面加载会更快.更少的图片,脚本和flash文件也就意味着更少的HTTP请求数,这就是提高页面加载速度的最佳方法之一.

某些图形化的CSS3技术,比如渐变,能够减少HTTP请求,但可能要求浏览器执行更多的工作来渲染,这就可能导致浏览器响应缓慢,并减少可用性.

因此不要随意滥用复杂的特性,并要完整地测试实现的那些特效.

有时候成功的使用CSS3最主要的阻碍反而不是浏览器的支持度,而是来自于客户或者老板的犹疑.

常见的CSS属性的浏览器前缀:

  1. -ms- Trident Internet Explorer

  2. -moz- Mozilla Firefox

  3. -o-* Presto Opera,Opera Mobile,Opera Mini,NIntendo wii浏览器

  4. -webkit- Webkit Safari,Safari IOS,Chrome,Android浏览器

使用前缀时候,应该总是把包含无前缀的属性并将它放在最后.

制作CSS气泡

  1. CSS的一些特性

word-wrap: 用来容纳溢出的文本

border-radius: 用来实现圆角

HSLA色彩模式: 用来创建半透明的背景

linear-gradient方法: 用来实现背景渐变

box-shadow: 用来实现物件的阴影

text-shadow: 用来实现文字的阴影

transform: 用来旋转物件

word-wrap早在IE5.5就支持这种属性.该属性实际上由微软创造,后来被W3C接受,其他浏览器都支持.

CSS3实现圆角只需要一个border-radius: 20px;
该语法可以被Opera,Chrome,Safari和IE9支持.Firefox和Safari4以及之前的版本则分别使用: -moz-border-radius和-webkit-border-radius.
IE8以及以前版本则会简单的忽略该属性,直接显示直角.
如果要实现兼容IE8问题,可以使用如下的脚本:

1. [PIE] (http://css3pie.com "PIE")

2. Curved-Corner(http://code.google.com/p/curved-corner)

3. IE-CSS3(http://fetchak.com/id-css3)

4. DD_roundies(http://dillerdesign.com/experi-ment/DD_roundies)

关键你要权衡的是IE用户实现圆角所花费的额外工作量和造成的性能是否值得.

使用CSS3来制作三角形: 

<div class="sgsheg"></div>

<style type="text/css">
    .sgsheg {
        border-color: red green blue orange;
        border-style: solid;
        border-width: 20px;
    `   width: 0;
        height: 0;
    }
</style>

border-color代表的是”上””右””下”左”

实现三角形的策略就是让其中的三个边框颜色设置为透明.(transparent),那么上面的就是一个三角形了.

代码如下:

<style type="text/css">
    .sgsheg {
        border-color: transparent green transparent transparent;
        border-style: solid;
        border-width: 20px;
        width: 0;
        height: 0;
    }
</style>

设计一个纯CSS三角形的要素是: 将一个元素的width和height设置为0,然后为它设置较粗的边框,并将其他的边框颜色设置为透明.你还可以为相邻边框设置边框高度来改变三角形的角度.

CSS3将伪元素使用了两个冒号而伪类则保持为单冒号.只要你愿意,你可以继续使用单冒号的版本,它可以兼容.IE8以及之前的版本不支持双冒号的方法,所以还是继续使用单冒号为好.

空格的HTML实体写作 ,可直译为”不折行间断”.它插入不会导致内容被浏览器换行.

.caption:before{

.blockquote:after {

CSS3的色彩模式:

  1. RGBA
    RGBA指的是: 红色,绿色,蓝色,Alpha透明度.(Red-Green-Blue-Alpha)

  2. HSLA
    HSLA则代表”色调,饱和度,亮度和Alpha透明度”(Hue-Saturation-Lightness-Alphs).

  3. Hex #A6DADC
    即颜色值的十六进制形式

对于CSS3上面的渐变性的兼容方案:

  1. 使用图片(创建半透明的PNG渐变图片),并设置为blockquote的背景图平铺.

缺点: 创造并使用图片会抵消你使用CSS3来实现渐变所获得的效率提升.Firefox3.
6不会加载并不需要的那张图片,但是chrome和safari会加载那张图片.

虽然在Firefox上保留了CSS3渐变的效率优势,但是在Safari却没有.虽然你依旧享受到了由CSS3渐变带来的其他优势,但效果却是其中最重要的优势.

关于渐变的支持,IE是不支持的,firefox需要-moz-,Safari需要-webkit前缀,Chrome支持,也需要-webkit-前缀

譬如实现一个颜色渐变的导航栏:

.blockquote {
    border-top: 1px solid red;
    min-height: 40px;
    margin: 0 0 0 112px;
    padding: 10px 15px 15px 15px;
    -moz-border-radius: 20px;
    -webkit-border-radius:  20px;
    border-radius: 20px;
    background-color: hsla(169, 41%, 76%, .5);
    background-image: -moz-linear-gradient(hsla(0,0%,-100%,.6), hsla(0,0%,100%,0) 30px);    //针对firefox内核
    background-image: -webkit-gradiennt(linear, 0 0, 0 30, from(hsla(0,0%,100%,.6))
                to (hsla(0,0%, 100%,0))
                );  //针对webkit内核
    word-wrap: break-word;
}

background-image 制定了渐变的起始色,(hsla(0,0,100%,.6),终止色(hsla(0,0%,100%,0)和终止色的位置(30px).

关于渐变:

1. 实现渐变只需要指定两种颜色,其余的值都是可选的.如果不指定那么就是默认值.
2. 可以在函数的第一个值里指定渐变的起始点,取值可以是关键词或数字.
3. 指定了启示位置后,你就是可以接着指定渐变的角度.
4. 每个颜色区间都包含一个颜色值.

本质上说,既然background-image属性中的渐变是浏览器生成的图片,那么可以使用其他CSS属性来进行控制,就像对待其他图片那样.

比如可以使用background-size来控制渐变的尺寸,或用background-repeat来控制是否将其平铺.

是使用图片来制作渐变,还是在不支持版本上不显示渐变,这个需要权衡

box-shadow

box-shadow盒装阴影.
譬如:

-moz-box-shadow: 1px 1px 2px hsla(0,0%,0%,.3);
-webkit-box-shadow: 1px 1px 2px hsla(0,0%,0%, .3);
box-shadow: 1px 1px 2px hsla(0,0%,0%,.3);

无前缀的box-shadow适应于IE9和Opera.
每个属性的第一个1px表示相对于盒容器的水平偏移,它告诉浏览器将阴影从盒容器向右移动1像素.第二个值1px则是浏览器从盒模型的边界向右移动1像素.

IE8以及以前的版本不支持box-shadow属性,正如渐变一样.你可以使用IE的滤镜来实现这种方法.

text-shadow

text-shadow: 文字阴影,首先不是所有的文本都应该加上阴影效果的,影响到页面可用性的反面教材不胜枚举.

text-shadow: 1px 1px 1px hsla(0, 0%, 100%, .7);

text-shadow是文本模块的一部分
text-shadow原属于CSS2,CSS2.1中移除了它,CSS3中又回来了.

当然如果IE要实现这效果,那么就要使用滤镜了.而非IE的浏览器,Opera,Chrome,Firefox和Safari都支持该属性.但是IE是不支持该属性的,针对IE的兼容性方案是你可以使用IE的滤镜.

transform

transform变形,translate是3D变形的一部分,你可以在(http://www.w3.org/tr/css3-3d-transform) 它,也可以在2D变形上找到它(http://www.w3.org/tr/css3-2d-transform) 上找到它. 所有的2D函数也包含3D规范,因此你可以只用查阅3D规范即可.

(2D变形)浏览器对变形的支持:

  1. IE不支持,

  2. Opera支持,需要-o-前缀,Firefox支持,但是需要-moz-,chrome和Safira支持,但需要-webkit-前缀

(3D变形)浏览器的支持:

  1. IE,Opera,Firefox,Chrome都不支持

  2. Safari支持,需要-webkit-前缀

代码:

.sgsheg {
    -moz-transform: rotate(-5deg);
    -o-transform: rotate(-5deg);
    -webkit-transform: rotate(-5deg);
    transform: rotate(-5deg);   //没有任何浏览器支持直接使用transform属性,为了以后的兼容性
}

当然你不会对所有的页面文本使用倾斜和旋转效果,那样绝对会影响可读性.但少量的使用变形会替代需要用图片或flash才能完成的的文字效果,能确实提高效果.
